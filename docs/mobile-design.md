# 手机端设计决策

> 状态：待定（Phase 6 前决策）

## 核心问题：一套页面响应式 vs. 独立手机页

### 方案 A：单页面响应式（改造 v3.html）
优点
- 一套代码维护
- 数据层、状态管理天然一致
- 一处改动全端生效

缺点
- v3.html 已接近 8000 行，再加响应式分支会更难维护
- 桌面的密集信息流在手机上几乎都要重排
- 很多桌面交互（悬浮提示、右键菜单）在触摸端无意义

### 方案 B：独立 mobile.html + shared/ 共用逻辑
优点
- 手机端可以只保留高频操作（打卡、看今日任务、记笔记）
- 移动端布局从零设计，不被桌面语义束缚
- 性能更好，首屏更小

缺点
- 两套 UI 要各自维护
- 功能对齐时有重复劳动

## 倾向

**初步倾向方案 B**，理由：
- 手机上不会做博士论文深度编辑，功能可以精简
- 桌面版保留不动，不引入响应式 bug 风险
- shared/data-layer.js 保证数据读写一致

## 手机端优先功能（MVP）

1. 早晨打卡（起床时间）
2. 查看今日任务 / 会议
3. 快速写一条反思
4. 查看本周出勤天数
5. 晚间打卡（睡觉时间）

## 手机端暂不做

- 写作章节编辑（字数多、键盘体验差）
- 投稿管理复杂表单
- 文献数据看板
